# 端口自动探测与服务管理说明

## 背景

Mac 桌面端开发时涉及 **4 个本地服务**，之前使用固定端口，当端口被占用时会导致启动失败或冲突。

```
Vite Dev Server   → 5173
Netease API       → 38761
Unblock Proxy     → 38762
Unblock Match     → 38763
```

端口探测功能让这些服务在端口被占用时自动切换到可用端口，且各服务之间正确感知彼此的地址。

## 各模块详情

### 1. `electron/port-manager.js` — 端口探测

**核心函数：**

- `isPortAvailable(port, host?)` — 用 `net.createServer().listen()` 测试端口是否可用，返回 `boolean`
- `pickPort(basePort, maxTries?)` — 从 basePort 开始递增查找可用端口
- `resolveServicePorts(preferred?, maxTries?)` — 依次解析 4 个服务端口，保证无冲突

**冲突解决策略：**

1. 从基准端口开始，逐 +1 探测可用端口（最多试 20 次）
2. Vite 优先探测（需要已知端口做 wait-on）
3. 已分配端口会被加入 `usedPorts` Set，后续服务跳过

```js
// 使用示例
import { resolveServicePorts } from './port-manager.js';
const ports = await resolveServicePorts();
// → { vite: 5173, api: 38761, unblockProxy: 38762, unblockMatch: 38763 }
```

### 2. `electron/serviceManager.js` — 服务管理

**能力：**

- `startAllServices(ports, skipUnblock?)` — 启动 3 个后端进程，返回子进程引用
- `waitApiReady(apiBaseUrl, timeoutMs?)` — 轮询 API `/banner` 端点，等待就绪
- `killAllServices(children)` — 终止所有子进程（SIGTERM）

**启动的三个服务：**

| 服务 | 启动命令 | 说明 |
|------|----------|------|
| Netease API | `node ${entrypoint}` （`ELECTRON_RUN_AS_NODE=1`） | 解析 `app.asar.unpacked` 或 `node_modules` 下的入口 |
| Unblock Proxy | `node app.js -p PORT -o bodian kugou migu qq bilibili -s` | 正向代理，拦截 Netease CDN 请求 |
| Unblock Match | `node server/unblock-match-server.mjs` | 独立匹配服务，供前端 `/unblock-api` 调用 |

**Netease API 入口点解析逻辑：**

1. 优先查找打包后的 `app.asar.unpacked/node_modules/@neteasecloudmusicapienhanced/api/`
2. 回退到开发目录的 `node_modules/@neteasecloudmusicapienhanced/api/`
3. 读取 `package.json` 的 `main`/`module` 字段确定入口文件

### 3. `electron/main.js` — 主进程集成

**启动流程：**

```
main.js
  │
  ├── 检查 SERVICE_PORTS 环境变量
  │     ├── 有 → 使用外部已探测端口（开发模式）
  │     └── 无 → 调用 port-manager 自动探测（生产打包模式）
  │
  ├── startAllServices(ports, isDev)
  │     ├── 开发模式：只启动 Netease API（unblock 由 orchestrator 负责）
  │     └── 生产模式：启动全部 3 个服务
  │
  ├── waitApiReady(apiBaseUrl, timeout)
  │     ├── 成功 → 创建主窗口
  │     └── 失败 → 显示错误页面
  │
  └── createMainWindow(ports)
        └── 把端口通过 --service-ports 参数传给 preload
```

**端口传递链路：**

```
resolveServicePorts()
    │
    ▼
startAllServices(ports)     ← 传环境变量给子进程
    │
    ▼
createMainWindow(ports)
    │
    ├── additionalArguments: ['--service-ports={...}']
    ▼
preload.js
    │
    ├── process.argv 解析 --service-ports
    ▼
contextBridge.exposeInMainWorld('appEnv', {
  apiPort, apiBaseUrl, unblockProxyUrl, unblockMatchUrl
})
    │
    ▼
src/api/client.ts
  → window.appEnv.apiBaseUrl      （API 请求地址）
  → window.appEnv.unblockProxyUrl （传递给 match 的代理地址）
```

### 4. `electron/preload.js` — 端口注入

从 `process.argv` 中解析 `--service-ports` JSON，通过 `contextBridge` 安全暴露给渲染进程：

```js
// 渲染进程访问
window.appEnv.apiPort          // Netease API 端口
window.appEnv.apiBaseUrl       // http://127.0.0.1:38761
window.appEnv.unblockProxyUrl  // http://127.0.0.1:38762
window.appEnv.unblockMatchUrl  // http://127.0.0.1:38763
```

### 5. `scripts/start-desktop.mjs` — 开发总控

**职责：**

1. 杀残留进程（固定端口列表，简化实现）
2. 调用 `resolveServicePorts()` 探测端口
3. 构造各个子进程的环境变量（`VITE_*`、`PORT`、`SERVICE_PORTS`）
4. 依次启动 Vite、unblock-proxy、unblock-match
5. 3 秒后启动 Electron
6. 注册 SIGINT/SIGTERM 清理回调

**环境变量传递：**

```
SERVICE_PORTS → JSON.stringify({api, unblockProxy, unblockMatch})
  │
  ├── Vite 环境:
  │     VITE_API_PROXY_TARGET     = http://127.0.0.1:{api}
  │     VITE_UNBLOCK_MATCH_TARGET = http://127.0.0.1:{unblockMatch}
  │     VITE_DEV_SERVER_URL       = http://localhost:{vite}
  │
  ├── Electron 环境:
  │     VITE_DEV_SERVER_URL       = http://localhost:{vite}
  │
  ├── Unblock Match 环境:
  │     PORT             = {unblockMatch}
  │     UNBLOCK_SOURCES  = bodian,kugou,migu,qq,bilibili
  │     (注意：不设置 UNBLOCK_PROXY_URL，直连 HTTPS 避免 CONNECT 问题)
  │
  └── Unblock Proxy 环境:
        ENABLE_FLAC = true
```

### 6. `vite.config.ts` — 动态代理

从环境变量读取代理目标，启动时由 `start-desktop.mjs` 注入：

```ts
const proxyTarget = env.VITE_API_PROXY_TARGET || 'http://127.0.0.1:38761';
const unblockMatchTarget = env.VITE_UNBLOCK_MATCH_TARGET || 'http://127.0.0.1:38763';
```

代理规则：

| 路径 | 目标 |
|------|------|
| `/api/*` | `{VITE_API_PROXY_TARGET}`（Netease API） |
| `/unblock-api/*` | `{VITE_UNBLOCK_MATCH_TARGET}`（Match Server） |
| `/dl-proxy?url=...` | Vite 中间件，透传音频流（添加 CORS 头） |

### 7. `server/unblock-match-server.mjs` — 端口支持

通过环境变量 `PORT` 指定监听端口（原为固定 38763）：

```js
const PORT = Number(process.env.PORT) || 38763;
server.listen(PORT, '0.0.0.0', ...);
```

## 开发模式与生产模式

### 开发模式 (`npm run dev:desktop`)

```
start-desktop.mjs  ← 总控脚本
  ├── 探测端口
  ├── 启动 vite / unblock-proxy / unblock-match
  ├── 设置 SERVICE_PORTS 环境变量
  └── 启动 Electron
        └── main.js 检测到 SERVICE_PORTS → 跳过端口探测，直接使用
              └── startAllServices(ports, skipUnblock=true)
                    只启动 Netease API，不启动 unblock（已由 orchestrator 启动）
```

### 生产模式（打包后）

```
用户启动 Electron
  └── main.js SERVICE_PORTS 不存在 → 自动探测端口
        └── startAllServices(ports, skipUnblock=false)
              启动全部 3 个服务
```

## 常见问题

### Q: 端口探测失败

确保没有其他进程占用了基准端口 ~ 基准端口+20 的范围。默认基准端口：

| 服务 | 基准端口 |
|------|----------|
| Vite | 5173 |
| API | 38761 |
| Unblock Proxy | 38762 |
| Unblock Match | 38763 |

### Q: 为什么 match server 不设 UNBLOCK_PROXY_URL？

`@unblockneteasemusic/server` 的匹配函数在 `global.proxy` 被设置时，所有 HTTPS 请求会通过该代理的 CONNECT 隧道。本地 unblock proxy 的 CONNECT 隧道对 `music.163.com` 的 TLS 握手失败（`SSL_ERROR_SYSCALL`），导致匹配直接挂掉。因此 match server 直连 HTTPS，不经过 unblock proxy。

### Q: 为什么 fee 探测不加 proxy 参数？

同上，API 服务器若配置了 `proxy` 参数，其对 `music.163.com` 的 HTTPS 请求会通过 unblock proxy 的 CONNECT 隧道，TLS 握手失败导致探测异常。Fee 探测直连判断官方可播性，音源替换由独立的 match server 负责。
