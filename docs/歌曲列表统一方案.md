# 歌曲列表渲染统一方案

## 1. 现状

6 个页面 / 组件的歌曲列表实现互不相同：

| 页面 | 渲染方式 | 行列数 | 播放按钮 | SongActions | 封面 | 数据源 |
|---|---|---|---|---|---|---|
| **PlaylistDetailPage** | 自有虚拟滚动（ROW_HEIGHT=68px, overscan=15, ResizeObserver） | `40px 52px 1fr auto` | PlayPauseButton | 有（收藏/下一首/加到歌单/评论/下载） | 52×52 | `playlist.value?.tracks`（分页懒加载） |
| **AlbumDetailPage** | 普通 v-for | 同上 | PlayPauseButton | 有（同上） | 52×52 | `album.value?.songs`（一次性） |
| **ArtistDetailPage** | 普通 v-for | 同上 | PlayPauseButton | 有（同上） | 52×52 | `topSongs`（一次性） |
| **PodcastDetailPage** | 普通 v-for | `40px 72px minmax(0,1fr)` **3列** | PlayPauseButton（emit 而非直接播放） | **无** | **72×72** | `props.items`（父组件传入，反向排序） |
| **SearchPage** | 普通 v-for | `40px 52px 1fr auto`（第4列空） | PlayPauseButton | **无** | 52×52 | `searchResult` API 响应 |
| **RankPanel** | 无歌曲行 — 卡片内嵌前3首预览 | `18px 1fr`（纯文本） | HoverPlayButton（卡片级，非行级） | **无** | 无行级封面 | `getToplistDetail` |

## 2. VirtualSongList 的位置

`VirtualSongList.vue` 是完整的虚拟滚动组件（ROW_HEIGHT=38px, overscan=15, 支持选择模式/右键菜单/排序列），但**仅被本地音乐页面使用**：

- `LocalSongsPage.vue`
- `LocalArtistsPage.vue`
- `LocalAlbumsPage.vue`
- `LocalFoldersPage.vue`

所有在线详情页都没用 VirtualSongList。PlaylistDetailPage 有自己的虚拟滚动实现（ROW_HEIGHT 68px vs 38px）。

## 3. 约束条件

### 3.1 不可忽略的差异

1. **PlaylistDetailPage 的虚拟滚动是硬需求** — 歌单可能 1000+ 首，必须虚拟滚动。但如果换用 VirtualSongList，ROW_HEIGHT 从 68px 变 38px 意味着大量样式调整。

2. **PodcastDetailPage 是结构异常值** — 3 列布局、72px 封面、无 SongActions、播放通过 emit 代理、反向排序、内嵌状态徽章。强行套用 4 列模版会破坏其设计。

3. **SearchPage 缺失 SongActions** — 搜索结果行没有收藏/评论/下载按钮。这是**产品决策**而非技术遗漏。

4. **RankPanel 完全不同的范式** — 没有歌曲行，是卡片发现页，不适用于行级统一。

### 3.2 VirtualSongList 的兼容性

VirtualSongList 使用 7 列网格（`grid-template-columns: 28px 32px 36px minmax(160px, 2fr) minmax(120px, 1fr) minmax(120px, 1fr) 40px 60px`），与详情页基线（`40px 52px 1fr auto`）完全不同。这意味着：

- 详情页接入 VirtualSongList ≈ 大规模重写
- 或者改造 VirtualSongList 支持灵活列配置

## 4. 可选路径

### 路径 A：提取 useSongRowConfig composable（低风险）

为 PlaylistDetailPage、AlbumDetailPage、ArtistDetailPage 三个最相似的页面提取共享逻辑：

```ts
// composables/useSongRowConfig.ts
export function useSongRowConfig(options: {
  tracks: Ref<any[]>
  onPlay: (index: number) => void
  showSongActions?: boolean  // 默认 true，SearchPage 传 false
}) {
  // 返回共享的：
  // - isCurrentTrack(songId) 判断
  // - onDblClick 处理（跳过播放按钮区域）
  // - onPlayEmit 处理
  // - song-item--playing class 绑定
}
```

**优点**：风险最低、改动最小、不改变 UI  
**缺点**：不解决虚拟滚动的重复实现问题  
**改动范围**：3 个详情页（PlaylistDetail/AlbumDetail/ArtistDetail）

### 路径 B：改造 VirtualSongList 支持详情页模式（中风险）

为 VirtualSongList 增加 `variant="detail"` 模式：
- `detail` 模式使用详情页基线列（`40px 52px 1fr auto`）、68px 行高、无 checkbox、无排序列
- PlaylistDetailPage 的 ROW_HEIGHT 从 68px → 68px（不变），但虚拟滚动实现从自有一份改为复用
- 其他详情页的 v-for 也可替换为 `<VirtualSongList variant="detail">`

**优点**：统一虚拟滚动实现  
**缺点**：VirtualSongList 更复杂；PlaylistDetailPage 的分页懒加载逻辑需迁移  
**改动范围**：VirtualSongList + 5 个详情页

### 路径 C：不做统一，仅规范化（最务实）

当前除 PodcastDetailPage 外，所有详情页已共享：
- 相同的 4 列布局（`detail-page.css` 的 `.song-item`）
- 相同的 PlayPauseButton 播放按钮
- 相同的 `song-item--playing` 高亮

差异主要在：
- 虚拟滚动 vs v-for（技术上允许的合理差异 — 歌曲数量不同）
- SongActions 有无（产品差异 — SearchPage/PodcastDetailPage 无收藏）

**结论**：在 `detail-page.css` 已经统一了行级样式的前提下，行级 JS 逻辑（双击、正在播放判断）差异小且合理。不建议为统一而统一。

## 5. 决策

**采用路径 A + C 的组合**，路径 A 已落地完成。

## 6. 路径 A 落地记录

### 6.1 已创建 `src/composables/useSongRowConfig.ts`

```ts
export function useSongRowConfig(playFn: (index: number) => Promise<void> | void) {
  function isCurrentTrack(song: any): boolean {
    return isCurrentTrackUtil(Number(song?.id || 0), Number(playerStore.currentSongId || 0));
  }

  function onSongItemDblClick(event: MouseEvent, index: number): void {
    const target = event.target as HTMLElement | null;
    if (target?.closest('button, a, input, select, textarea, [role="button"]')) return;
    void playFn(index);
  }

  return { isCurrentTrack, onSongItemDblClick };
}
```

接收各页面自己的播放回调 `playFn`，返回统一的 `isCurrentTrack` 和 `onSongItemDblClick`。

### 6.2 已接入页面

| 页面 | 接入方式 | 删除代码 |
|---|---|---|
| **PlaylistDetailPage** | `useSongRowConfig(playOne)` | 私有 `isCurrentTrack()`、`onSongItemDblClick()` |
| **AlbumDetailPage** | `useSongRowConfig(playOne)` | 同上 |
| **ArtistDetailPage** | `useSongRowConfig(playSong)` | 同上 |

### 6.3 未接入页面（合理保留）

| 页面 | 原因 |
|---|---|
| **PodcastDetailPage** | 结构异常值（3 列、72px 封面、无 SongActions、emit 代理），强行统一会破坏设计 |
| **SearchPage** | 无 SongActions 是产品决策；如需补上，单独处理 |
| **RankPanel** | 无歌曲行，卡片发现页范式完全不同 |

### 6.4 不做

1. **不做组件级合并** — 行布局已由 `detail-page.css` 统一，虚拟滚动 vs v-for 差异合理（大歌单需虚拟滚动，小专辑不需要）。
2. **VirtualSongList 重构** — 单独项目，需先统一 ROW_HEIGHT（68px vs 38px）再决定策略。

### 7.4 统一 `.card` 样式到 `theme.css`

MvPanel 的 `.card` 私有规则（`overflow: hidden; cursor: pointer; transition; :hover` 上浮动效）与全局卡片基类存在重复。

**改动**：

1. `theme.css` 的 `.card` 基类新增 `overflow: hidden`（所有卡片容器通用）
2. `theme.css` 新增 `.card-clickable` 变体（`cursor: pointer` + `transition` + `:hover` 上浮）
3. MvPanel scoped 中删除 `.card` 和 `.card:hover` 私有规则，模板改为 `class="card card-clickable"`
4. HomePanel / SearchPage 无需改动 — 它们使用 `.card` 作为非交互容器，自动继承 `overflow: hidden`

| 文件 | 改动 |
|---|---|
| `theme.css` | `.card` + `overflow: hidden`；新增 `.card-clickable` + `.card-clickable:hover` |
| `MvPanel.vue` | 模板 `card` → `card card-clickable`；删除 scoped `.card`/`.card:hover` |
