# 统一架构说明

本文档记录当前首页、每日推荐、云盘、用户页歌单详情页已经统一后的数据与展示架构。

## 1. 目标

当前项目的目标是把不同来源的音乐内容统一成同一种“歌单详情页体验”：

- 每日推荐
- 云盘歌曲
- 普通歌单

它们在 UI 上尽量保持一致，只在数据来源上保留差异。

---

## 2. 统一后的核心模型

项目里现在统一使用 `PlaylistLike` 作为详情页数据模型。

统一字段如下：

- `name`
- `coverImgUrl`
- `description`
- `creator`
- `trackCount`
- `tracks`

这个模型会被 `PlaylistDetailPage` 直接消费。

### 数据模型示意

```ts
{
  name: string,
  coverImgUrl: string,
  description?: string,
  creator?: {
    nickname?: string,
    avatarUrl?: string,
  },
  trackCount: number,
  tracks: Song[]
}
```

---

## 3. 统一入口与来源

### 3.1 普通歌单

- 数据来源：`/playlist/detail`
- 直接得到歌单对象
- 进入 `PlaylistDetailPage` 时通常不需要额外注入

### 3.2 每日推荐

- 数据来源：`/recommend/songs`
- 返回的是歌曲列表，不是歌单对象
- 先通过适配器构造成 `PlaylistLike`
- 再注入 `PlaylistDetailPage`

### 3.3 云盘

- 数据来源：`/user/cloud`、`/user/cloud/detail`
- 先把云盘歌曲整理成统一歌曲列表
- 再构造成 `PlaylistLike`
- 再注入 `PlaylistDetailPage`

---

## 4. 公共适配层

项目里新增了 `src/utils/playlistLike.ts`，用于把不同来源整理成统一模型。

### 主要函数

- `createPlaylistLike(input)`
- `normalizeRecommendSongsPlaylist(songs, nickname)`
- `normalizeCloudPlaylist(songs, profile)`

### 作用

这些函数负责：

- 统一歌单标题
- 统一封面图
- 统一描述文案
- 统一 creator 信息
- 统一 trackCount
- 统一 tracks 列表

这样后续新增一个“歌单样式”入口，只需要先整理数据，再交给 `PlaylistDetailPage`。

---

## 5. 统一详情页

当前所有歌单样式内容最终都进入同一个详情页组件：

- `src/components/PlaylistDetailPage.vue`

### 这个组件现在支持两种模式

#### 5.1 真歌单模式

- 传入 `playlistId`
- 组件内部请求 `/playlist/detail`
- 渲染真实歌单详情

#### 5.2 注入模式

- 传入 `injectedPlaylist`
- 组件直接渲染注入数据
- 不再请求真实歌单接口

### 额外支持

- **吸顶连续进度**（`DetailStickyHeroHeader` 内置）
- 不再需要手动传 `scrollHostSelector`
- 自动向上查找最近的可滚动父节点作为滚动容器
- 所有详情页吸顶行为统一：连续 `--sticky-progress` 驱动，无二值切换
- **布局**：基础态 actions 在封面下方（`.hero-media-col` 竖向布局），吸顶态标题 flex 拉伸 + 按钮右侧绝对定位
- **文档流恒定**：`min-height` 由 ResizeObserver 测量内层 `.page-hero-header` 高度自动设置，防止自膨胀循环
- **语义结构**：`DetailStickyHeroHeader` 外层 `.playlist-detail-header-wrap` 是唯一语义 `header`，同时也是吸顶控制节点；内部复用 `PageHeroHeader` 时传入 `tag="div"`，避免产生两层 `header`
- **头部-内容区分隔线**：基础态底部弱渐隐分割线，吸顶态自动隐藏

---

## 6. 首页架构

首页由 `HomePanel.vue` 负责。

### 当前首页上方两个卡面

#### 第一个卡面：每日推荐

- 数据来源：`/recommend/songs`
- 通过 `normalizeRecommendSongsPlaylist()` 变成统一模型
- 点击后进入统一详情页/播放链路

#### 第二个卡面：私人雷达

- 保持原来的私人雷达逻辑
- 仍然使用原有数据接口
- 与每日推荐保持 UI 风格一致，但数据源不同

### 首页播放逻辑

首页每日推荐的播放已经接入公共播放工具：

- `src/utils/trackPlayback.ts`

统一做到：

- 点击歌曲时按需获取 URL
- 再播放
- 不再在首屏批量请求播放地址

---

## 7. 用户页架构

用户页由 `UserPanel.vue` 负责。

### 当前用户页内容

- 创建歌单
- 收藏歌单
- 收藏专辑
- 云盘

### 当前统一方式

#### 歌单列表

- 左侧列表选择歌单
- 右侧通过 `PlaylistDetailPage` 展示

#### 云盘

- 云盘歌曲先整理成 `cloudPseudoPlaylist`
- 再注入 `PlaylistDetailPage`
- 表现为“伪歌单详情页”

### 吸顶状态同步

`DetailStickyHeroHeader` 自动检测右侧详情区域的滚动容器（`.detail-panel`），无需外部传入 `scrollHostSelector`。

吸顶行为由连续滚动进度 `--sticky-progress`（0→1）驱动，所有详情页（Playlist、Album、Artist、Podcast、User）统一。

---

## 8. 播放架构

项目新增了统一播放工具：

- `src/utils/trackPlayback.ts`

### 主要方法

- `resolveTrackUrl(track)`
- `playTrackWithResolvedUrl(playerStore, track, index)`

### 作用

- 不在首屏批量请求播放地址
- 只有真正播放时才取 URL
- 首页、歌单详情页、云盘都复用同一逻辑

### 当前接入点

- 首页每日推荐
- `PlaylistDetailPage`
- 用户页云盘

---

## 9. 性能优化原则

当前统一后的优化原则是：

### 9.1 首屏先展示

只渲染：

- 标题
- 封面
- 描述
- 列表

### 9.2 播放时再补齐 URL

不要在首屏把每首歌的播放地址都请求完。

### 9.3 统一展示，局部差异

- 展示层统一
- 数据来源差异保留
- 播放行为统一

---

## 10. 当前架构总结

现在项目已经从“多个入口各自实现”收敛为：

1. **统一数据模型**
   - `PlaylistLike`

2. **统一详情页**
   - `PlaylistDetailPage`

3. **统一播放工具**
   - `trackPlayback.ts`

4. **统一适配层**
   - `playlistLike.ts`

5. **统一吸顶进度**
   - `DetailStickyHeroHeader` 内置连续 `--sticky-progress`
   - 自动检测滚动容器，无需外部传参
   - 内联样式覆盖所有详情页二值 CSS，保证行为一致
   - 基础态 actions 在封面下方，吸顶态标题拉伸 + 按钮右侧绝对定位
   - `min-height` 由 ResizeObserver 测量内层高度设定，防止自膨胀
   - 基础态底部弱分割线 + `padding-bottom: var(--space-2)` 呼吸间距

这意味着后续新增“歌单样式内容入口”时，只需要：

- 先把原始接口数据适配成 `PlaylistLike`
- 再交给 `PlaylistDetailPage`
- 播放时走统一工具

---

## 11. 维护建议

### 推荐的新增方式

如果未来再加类似“歌单样式”的内容入口，建议按以下流程：

1. 拉取原始接口数据
2. 通过适配器转成 `PlaylistLike`
3. 注入 `PlaylistDetailPage`
4. 播放时使用 `playTrackWithResolvedUrl`

### 不建议的方式

- 给每种来源各写一套详情页
- 在首屏批量请求所有播放地址
- 让云盘、每日推荐、歌单分别维护不同 UI 结构

---

## 12. 结论

当前项目已经完成了“多入口统一成单一歌单体验”的收敛：

- **展示统一**
- **数据统一**
- **播放统一**
- **滚动吸顶统一**（连续进度驱动，自动容器检测，内联样式覆盖）

这套架构的核心就是：

> 不管数据来自哪里，最后都变成同一种歌单详情页对象，再交给同一个详情页渲染和播放。

## 13. 详情页架构统一化改动记录

### 13.1 背景变量统一

**问题**：`--cover-bg-url`（全局）与 `--cover-bg`（组件内）并存。

**方案**：统一使用 `--cover-bg-url`。

**改动文件**：
- `src/styles/detail-page.css`：`var(--cover-bg)` → `var(--cover-bg-url)`
- 6 个详情页 + `RankPanel.vue`：shellStyle 返回值从 `--cover-bg` 改为 `--cover-bg-url`

### 13.2 播客详情页加入 hero sticky

**问题**：`App.vue` 的 `isHeroStickyPage` 不包含 `podcast-detail`，播客详情页没有顶层 hero sticky 背景。

**改动**：`src/App.vue` 的 `isHeroStickyPage` 数组加入 `'podcast-detail'`。

### 13.3 吸顶状态逻辑抽取为 composable

**问题**：6 个详情页各自维护重复的吸顶状态判断逻辑（~35 行/页）。

**新建文件**：`src/composables/useDetailStickyState.ts`

**接口**：
```ts
interface UseDetailStickyStateOptions {
  scrollHostSelector?: string | (() => string | undefined);
  headerWrapSelector?: string;    // 默认 '.playlist-detail-header-wrap'
  enterThreshold?: number;        // 默认 12
  exitThreshold?: number;         // 默认 4
  safetyGap?: number;             // 默认 32
}

function useDetailStickyState(options?): { isSticky: Ref<boolean>, refresh: () => void }
```

**改动文件**：
- `PlaylistDetailPage.vue` / `AlbumDetailPage.vue` / `ArtistDetailPage.vue` / `PodcastDetailPage.vue` / `UserDetailPage.vue` / `LanguageDetailPage.vue`：删除重复逻辑，改为调用 composable

### 13.4 DetailStickyHeroHeader DOM 查询改为组件级

**问题**：使用 `document.querySelector('.playlist-detail-header-wrap')` 全局查询，多实例时可能查错。

**方案**：添加 `rootEl` ref 绑定到模板根元素，内部通过 `resolveEl()` 解析（支持组件实例 `$el` 和原生元素）。

**改动文件**：`src/components/DetailStickyHeroHeader.vue`

### 13.5 吸顶态布局修复

**问题 1**：吸顶态标题靠上，未垂直居中。

**根因**：`clip-path` 只显示顶部 ~80px，但 `align-items: center` 在整个元素高度（324px）内居中，内容实际在顶部。

**修复**：
- `.page-hero-header` 改用 `align-items: start`
- `.page-hero-header__content` 改用 `display: block`
- 增加 `padding-top: 22px`（标题约 36px → (80-36)/2 ≈ 22px）

**问题 2**：吸顶态按钮 `top: var(--space-3)` 不匹配新 padding-top。

**修复**：`.hero-actions-shell--under-cover` 的 `top` 改为 `22px`。

### 13.6 详情页背景 CSS 统一

**问题**：`detail-page.css` 的 `::before`（封面模糊背景）只覆盖 `.panel` 和 `.daily-page`，不覆盖各详情页根 class。

**修复**：
- `::before` 选择器扩展：加入 `.playlist-detail-page::before`、`.artist-detail-page::before`、`.user-detail-page::before`（`.podcast-detail-page` 已有 `playlist-detail-page` base class，自动覆盖）
- `::after` 渐隐过渡层已完全移除

**最终 CSS 结构**：
- `::before`：封面模糊背景渐变（`blur(24px) saturate(155%)`）→ 所有详情页共享
- 吸顶态：磨砂透明背景（`color-mix(in srgb, var(--bg-surface) 82%, transparent)` + `backdrop-filter: blur(10px)`）

### 13.7 吸顶态布局最终状态

- **背景**：磨砂透明（`color-mix(in srgb, var(--bg-surface) 82%, transparent)` + `backdrop-filter: blur(10px) saturate(132%)`）
- **垂直居中**：`padding-top: 22px`（标题 ~36px 居中于 ~80px 可见区域）
- **按钮对齐**：`.hero-actions-shell--under-cover` 的 `top: 22px` 匹配标题
- **grid 布局**：`.page-hero-header` 用 `align-items: start`，`.page-hero-header__content` 用 `display: block`
