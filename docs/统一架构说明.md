# 统一架构说明

本文档记录当前首页、每日推荐、云盘、用户页歌单详情页已经统一后的数据与展示架构。

## 1. 目标

当前项目的目标是把不同来源的音乐内容统一成同一种“歌单详情页体验”：

- 每日推荐
- 云盘歌曲
- 普通歌单

它们在 UI 上尽量保持一致，只在数据来源上保留差异。

---

## 2. 统一后的核心模型

项目里现在统一使用 `PlaylistLike` 作为详情页数据模型。

统一字段如下：

- `name`
- `coverImgUrl`
- `description`
- `creator`
- `trackCount`
- `tracks`

这个模型会被 `PlaylistDetailPage` 直接消费。

### 数据模型示意

```ts
{
  name: string,
  coverImgUrl: string,
  description?: string,
  creator?: {
    nickname?: string,
    avatarUrl?: string,
  },
  trackCount: number,
  tracks: Song[]
}
```

---

## 3. 统一入口与来源

### 3.1 普通歌单

- 数据来源：`/playlist/detail`
- 直接得到歌单对象
- 进入 `PlaylistDetailPage` 时通常不需要额外注入

### 3.2 每日推荐

- 数据来源：`/recommend/songs`
- 返回的是歌曲列表，不是歌单对象
- 先通过适配器构造成 `PlaylistLike`
- 再注入 `PlaylistDetailPage`

### 3.3 云盘

- 数据来源：`/user/cloud`、`/user/cloud/detail`
- 先把云盘歌曲整理成统一歌曲列表
- 再构造成 `PlaylistLike`
- 再注入 `PlaylistDetailPage`

---

## 4. 公共适配层

项目里新增了 `src/utils/playlistLike.ts`，用于把不同来源整理成统一模型。

### 主要函数

- `createPlaylistLike(input)`
- `normalizeRecommendSongsPlaylist(songs, nickname)`
- `normalizeCloudPlaylist(songs, profile)`

### 作用

这些函数负责：

- 统一歌单标题
- 统一封面图
- 统一描述文案
- 统一 creator 信息
- 统一 trackCount
- 统一 tracks 列表

这样后续新增一个“歌单样式”入口，只需要先整理数据，再交给 `PlaylistDetailPage`。

---

## 5. 统一详情页

当前所有歌单样式内容最终都进入同一个详情页组件：

- `src/components/PlaylistDetailPage.vue`

### 这个组件现在支持两种模式

#### 5.1 真歌单模式

- 传入 `playlistId`
- 组件内部请求 `/playlist/detail`
- 渲染真实歌单详情

#### 5.2 注入模式

- 传入 `injectedPlaylist`
- 组件直接渲染注入数据
- 不再请求真实歌单接口

### 额外支持

- `scrollHostSelector`
- 用于嵌入式场景下指定滚动容器
- 解决用户页右侧面板吸顶状态同步问题

---

## 6. 首页架构

首页由 `HomePanel.vue` 负责。

### 当前首页上方两个卡面

#### 第一个卡面：每日推荐

- 数据来源：`/recommend/songs`
- 通过 `normalizeRecommendSongsPlaylist()` 变成统一模型
- 点击后进入统一详情页/播放链路

#### 第二个卡面：私人雷达

- 保持原来的私人雷达逻辑
- 仍然使用原有数据接口
- 与每日推荐保持 UI 风格一致，但数据源不同

### 首页播放逻辑

首页每日推荐的播放已经接入公共播放工具：

- `src/utils/trackPlayback.ts`

统一做到：

- 点击歌曲时按需获取 URL
- 再播放
- 不再在首屏批量请求播放地址

---

## 7. 用户页架构

用户页由 `UserPanel.vue` 负责。

### 当前用户页内容

- 创建歌单
- 收藏歌单
- 收藏专辑
- 云盘

### 当前统一方式

#### 歌单列表

- 左侧列表选择歌单
- 右侧通过 `PlaylistDetailPage` 展示

#### 云盘

- 云盘歌曲先整理成 `cloudPseudoPlaylist`
- 再注入 `PlaylistDetailPage`
- 表现为“伪歌单详情页”

### 吸顶状态同步

用户页里嵌入 `PlaylistDetailPage` 时，显式传入：

- `scrollHostSelector=".detail-panel"`

用于让右侧详情区域使用正确的滚动容器计算 sticky 状态。

---

## 8. 播放架构

项目新增了统一播放工具：

- `src/utils/trackPlayback.ts`

### 主要方法

- `resolveTrackUrl(track)`
- `playTrackWithResolvedUrl(playerStore, track, index)`

### 作用

- 不在首屏批量请求播放地址
- 只有真正播放时才取 URL
- 首页、歌单详情页、云盘都复用同一逻辑

### 当前接入点

- 首页每日推荐
- `PlaylistDetailPage`
- 用户页云盘

---

## 9. 性能优化原则

当前统一后的优化原则是：

### 9.1 首屏先展示

只渲染：

- 标题
- 封面
- 描述
- 列表

### 9.2 播放时再补齐 URL

不要在首屏把每首歌的播放地址都请求完。

### 9.3 统一展示，局部差异

- 展示层统一
- 数据来源差异保留
- 播放行为统一

---

## 10. 当前架构总结

现在项目已经从“多个入口各自实现”收敛为：

1. **统一数据模型**
   - `PlaylistLike`

2. **统一详情页**
   - `PlaylistDetailPage`

3. **统一播放工具**
   - `trackPlayback.ts`

4. **统一适配层**
   - `playlistLike.ts`

5. **统一嵌入滚动处理**
   - `scrollHostSelector`

这意味着后续新增“歌单样式内容入口”时，只需要：

- 先把原始接口数据适配成 `PlaylistLike`
- 再交给 `PlaylistDetailPage`
- 播放时走统一工具

---

## 11. 维护建议

### 推荐的新增方式

如果未来再加类似“歌单样式”的内容入口，建议按以下流程：

1. 拉取原始接口数据
2. 通过适配器转成 `PlaylistLike`
3. 注入 `PlaylistDetailPage`
4. 播放时使用 `playTrackWithResolvedUrl`

### 不建议的方式

- 给每种来源各写一套详情页
- 在首屏批量请求所有播放地址
- 让云盘、每日推荐、歌单分别维护不同 UI 结构

---

## 12. 结论

当前项目已经完成了“多入口统一成单一歌单体验”的收敛：

- **展示统一**
- **数据统一**
- **播放统一**
- **滚动吸顶统一**

这套架构的核心就是：

> 不管数据来自哪里，最后都变成同一种歌单详情页对象，再交给同一个详情页渲染和播放。
