## 阶段 1：基础设施

### 1.1 `src/utils/requestDedup.ts` — 请求去重

- 基于 Promise Map，相同 key 同时 pending 时复用 Promise
- 请求完成后立即释放 Map entry
- 生命周期短，只在请求期间生效

### 1.2 `src/stores/apiCache.ts` — 缓存管理器

- 全局 `reactive()` 单例
- 内存 Map + LRU 淘汰 + TTL 检查
- 缓存分组配置：

| 分组 | 示例 key | TTL | 最大条数 |
|---|---|---|---|
| 实体 | `song:${id}`, `album:${id}`, `artist:${id}` | 30min | 500/200/200 |
| 列表 | `playlist:${id}`, `toplist`, `highquality` | 10min / 5min | 200/50/50 |
| 不缓存 | `search:`, 推荐类 | 0 | 0 |

- 用户 ID 隔离：key 自动追加 `@${userId}`（仅对含身份字段的 API）
- 暴露方法：`getCache`, `setCache`, `hasCache`, `clearAll`, `clearUserScopedCache`
- 桌面端预留后端接口 `setBackend(backend)`

### 1.3 `src/composables/useApiData.ts` — SWR 组合式

API：

```ts
function useApiData<T>(
  cacheKey: string | (() => string),
  fetcher: () => Promise<T>,
  options?: {
    ttl?: number
    dedupMs?: number
    enabled?: boolean | Ref<boolean>
    immutable?: boolean
    onSuccess?: (data: T) => void
    onError?: (err: Error) => void
  }
): {
  data: Ref<T | null>
  loading: Ref<boolean>
  error: Ref<string>
  refresh: () => Promise<void>
  mutate: (data: T) => void
}
```

行为：
1. 构造 `cacheKey`（追加 `@${userId}` 如需要）
2. 检查 dedupMap → 相同 key pending → 复用 Promise
3. 检查 apiCache → 命中且未过期 → `immutable?` 直接返回 : data+后台静默刷新
4. 未命中 → 发起请求 → 写入缓存 → 返回数据

**重要**：身份相关字段（liked, subscribed）不在缓存中，必须从 `userStore` 读取

### 1.4 用户态修改

`src/stores/user.ts`：
- `logout()` 中新增 `clearApiCache()`
- `loginWithCookie/restoreUidLogin` 成功后调用 `clearApiCache()`
- 不清 `unblock-cache`（音源结果与用户无关）

## 数据缓存策略速查：按数据更新频率

### 每日更新数据（每日推荐、私人雷达）

适用 `dailyTtl()` 辅助函数（计算到次日 6:00 的毫秒数）：

```ts
function dailyTtl(): number {
  const now = Date.now();
  const next = new Date();
  next.setDate(next.getDate() + 1);
  next.setHours(6, 0, 0, 0);
  return Math.max(next.getTime() - now, 60 * 60 * 1000);
}

// 使用方式（在 fetch 函数中）：
const cached = apiCache.get('daily:songs');
if (cached?.data) {
  // 直接使用缓存数据，跳过 API
  return;
}
// 请求成功后：
apiCache.set('daily:songs', data, dailyTtl());
```

### 每次刷新更新数据（私人 FM）

**不缓存**。每次组件挂载/刷新时重新请求，不检查 apiCache，也不写入。

```ts
// 直接请求，不走缓存
const { data } = await getPersonalFm(userStore.loginCookie || undefined);
```

### 缓存决策表

| 数据特征 | 策略 | 示例 |
|---|---|---|
| 每日更新一次（6:00） | `dailyTtl()`，缓存到次日 6:00 失效 | 每日推荐、私人雷达 |
| 用户操作变更 | 不缓存实体状态，从 `userStore` 读取 | 收藏/订阅/关注状态 |
| 每次刷新需最新 | 不缓存，每次直接请求 | 私人 FM |
| 极少变更 | 长 TTL（10-30min） | 歌曲/专辑/歌手详情 |
| 中等变更频率 | 短 TTL（3-5min） | 歌单列表、排行榜 |

## 阶段 4：桌面端增强（Electron 持久化）

### 4.1 CacheBackend 接口抽像

```ts
interface CacheBackend {
  get(key: string): CacheEntry | null
  set(key: string, entry: CacheEntry): void
  delete(key: string): void
  clear(): void
}
```

Web 端使用 `MemoryBackend`，桌面端使用 `ElectronFileBackend`。

### 4.2 ElectronFileBackend

- 内存 Map + 文件持久化（userData 目录下 `api-cache.json`）
- 应用启动时加载文件到内存
- 写入防抖 2s 合并多次写入
- 文件过大时自动裁减（保留最近 500 条）

### 4.3 preload 暴露 + platform 模块

- `electron/preload.js` 新增 IPC 通道
- `global.d.ts` 更新
- `src/utils/platform.ts` 导出 `cacheBackend`

### 4.4 配置注入

```ts
// useApiData 内部
const cacheBackend = platform.isDesktop
  ? electronFileBackend
  : memoryBackend
```

## 平台差异汇总

| 特性 | Web | Electron 桌面 |
|---|---|---|
| 缓存后端 | 纯内存 Map | 内存 + 文件持久化 |
| 实体 TTL | 30min | 30min（无差异） |
| 列表 TTL | 5-10min | 10-15min |
| 最大条数 | 全部 300 | 全部 800 |
| 持久化 | 无 | 支持 |
| 冷启动缓存 | 无 | 从文件恢复 |
| preload 需求 | 无 | 需要 |
